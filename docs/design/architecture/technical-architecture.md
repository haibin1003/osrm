### 1.2 技术选型对比

| 层面 | 选型 | 备选方案 | 选型理由 |
|------|------|---------|---------|
| **前端框架** | Vue 3 | React | 国内生态完善，学习成本低，与Spring Boot配合成熟 |
| **后端框架** | Spring Boot 3.4 | Quarkus/Micronaut | 生态最完善，团队熟悉度高，企业级首选 |
| **数据库** | PostgreSQL 15 | MySQL 8 | JSON支持更好，复杂查询性能优，符合SQL标准 |
| **缓存** | Redis 7 | Memcached | 支持持久化，数据结构丰富，集群方案成熟 |
| **对象存储** | MinIO | 阿里云OSS | 私有部署，S3兼容，成本可控 |
| **消息队列** | RabbitMQ | Kafka/RocketMQ | 可靠性高，管理界面友好，适合企业级应用 |
| **容器编排** | Kubernetes | Docker Swarm | 生态最完善，扩展性强，云原生标准 |

**RabbitMQ 集群配置建议：**

| 配置项 | 建议值 | 说明 |
|--------|--------|------|
| 集群节点数 | 3 节点 | 2 Disc + 1 RAM，或 3 Disc |
| 镜像队列模式 | exactly / 2 | 每队列2个镜像，平衡可用性与性能 |
| 内存阈值 | 0.6 | 超过60%内存使用率触发流控 |
| 磁盘阈值 | 50MB | 磁盘剩余低于50MB停止接收消息 |
| 消息持久化 | 是 | 重要事件必须持久化 |
| TTL | 7天 | 消息默认过期时间 |

**Spring Boot 集成配置：**

```yaml
spring:
  rabbitmq:
    addresses: rabbitmq-1:5672,rabbitmq-2:5672,rabbitmq-3:5672
    username: ${RABBITMQ_USER:osrm}
    password: ${RABBITMQ_PASS:secret}
    virtual-host: /osrm
    connection-timeout: 5000
    template:
      mandatory: true
      retry:
        enabled: true
        max-attempts: 3
        initial-interval: 1000ms
    listener:
      simple:
        acknowledge-mode: manual
        concurrency: 5
        max-concurrency: 20
        prefetch: 10
        default-requeue-rejected: false
```

**Redis + Lua 分布式限流实现：**

```lua
-- 令牌桶限流 Lua 脚本
local key = KEYS[1]
local rate = tonumber(ARGV[1])        -- 每秒产生令牌数
local capacity = tonumber(ARGV[2])    -- 桶容量
local requested = tonumber(ARGV[3])   -- 请求令牌数
local now = tonumber(ARGV[4])         -- 当前时间戳（毫秒）

local fill_time = capacity / rate
local ttl = math.floor(fill_time * 2)

-- 获取当前状态
local last_tokens = redis.call('hget', key, 'tokens')
local last_updated = redis.call('hget', key, 'timestamp')

if last_tokens == false then
    last_tokens = capacity
    last_updated = now
else
    last_tokens = tonumber(last_tokens)
    last_updated = tonumber(last_updated)
end

-- 计算新增令牌
local delta = math.max(0, now - last_updated)
local filled_tokens = math.min(capacity, last_tokens + (delta * rate / 1000))

-- 判断是否允许请求
local allowed = filled_tokens >= requested
local new_tokens = filled_tokens
if allowed then
    new_tokens = filled_tokens - requested
end

-- 更新状态
redis.call('hset', key, 'tokens', new_tokens)
redis.call('hset', key, 'timestamp', now)
redis.call('expire', key, ttl)

return allowed and 1 or 0
```

```java
// Spring Boot 限流组件
@Component
public class DistributedRateLimiter {

    @Autowired
    private StringRedisTemplate redisTemplate;

    private static final String LUA_SCRIPT = "..."; // 上面的Lua脚本

    public boolean tryAcquire(String key, int rate, int capacity, int requested) {
        DefaultRedisScript<Long> script = new DefaultRedisScript<>(LUA_SCRIPT, Long.class);
        Long result = redisTemplate.execute(
            script,
            Collections.singletonList("rate_limit:" + key),
            String.valueOf(rate),
            String.valueOf(capacity),
            String.valueOf(requested),
            String.valueOf(System.currentTimeMillis())
        );
        return result != null && result == 1;
    }
}
```

#### 3.4.2 熔断降级策略

| 熔断器 | 触发条件 | 熔断时长 | 恢复策略 | 降级行为 |
|--------|---------|---------|---------|---------|
| 服务熔断 | 错误率 > 60% | 30秒 | 半开测试 | 返回缓存/默认值 |
| 慢调用熔断 | 响应时间 > 5s | 60秒 | 渐进恢复 | 简化查询 |
| 异常熔断 | 连续异常 > 10次 | 60秒 | 手动恢复 | 服务不可用提示 |

**Resilience4j 熔断配置：**

```yaml
resilience4j:
  circuitbreaker:
    configs:
      default:
        failureRateThreshold: 60           # 失败率阈值
        slowCallRateThreshold: 50          # 慢调用阈值
        slowCallDurationThreshold: 5s      # 慢调用定义
        permittedNumberOfCallsInHalfOpenState: 5
        slidingWindowSize: 100
        minimumNumberOfCalls: 20
        waitDurationInOpenState: 30s       # 熔断持续时间
        automaticTransitionFromOpenToHalfOpenEnabled: true
    instances:
      storageService:
        baseConfig: default
      externalHarbor:
        baseConfig: default
        failureRateThreshold: 50           # 外部服务更严格
  timelimiter:
    configs:
      default:
        timeoutDuration: 5s
        cancelRunningFuture: true
```

```java
// 熔断降级示例
@Service
public class StorageService {

    @CircuitBreaker(name = "storageService", fallbackMethod = "fallbackUpload")
    @TimeLimiter(name = "storageService")
    public CompletableFuture<String> uploadFile(MultipartFile file) {
        return CompletableFuture.supplyAsync(() -> {
            // 上传逻辑
            return minioClient.upload(file);
        });
    }

    // 降级方法
    private CompletableFuture<String> fallbackUpload(MultipartFile file, Exception ex) {
        log.warn("Storage service unavailable, using fallback", ex);
        // 降级：保存到本地临时目录，稍后同步
        String tempPath = localTempStorage.save(file);
        return CompletableFuture.completedFuture("temp:" + tempPath);
    }
}
```
