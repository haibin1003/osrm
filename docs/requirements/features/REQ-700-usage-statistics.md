# 功能需求文档：使用统计看板

**功能编号**: REQ-700
**需求编号**: REQ-700 ~ REQ-799
**优先级**: P0 (Phase 2)
**创建日期**: 2026-03-25
**状态**: 需求设计阶段

---

## 1. 功能概述

### 1.1 背景
Phase 1 MVP 已完成软件包的录入、审核、发布、订购全流程。为了帮助企业了解开源软件的使用情况、优化软件资产管理决策，需要提供全面的使用统计视图。

### 1.2 目标
- 提供多维度的软件使用统计视图
- 支持按时间范围筛选统计 data
- 展示业务系统的软件依赖分布
- 识别热门软件包和使用趋势
- 为软件管理员和系统管理员提供数据支撑

### 1.3 业务范围

| 在范围内 | 不在范围内（后续版本） |
|---------|---------------------|
| 订购量趋势统计 | 实时在线使用统计 |
| 业务系统软件分布 | 版本级别详细使用统计 |
| 软件包热度排行 | 下载次数统计（需制品仓库对接） |
| 类型分布统计 | 许可证合规统计 |
| 自定义时间范围筛选 | 预测分析/智能推荐 |

## 3. 功能需求

### 3.1 统计概览卡片 (REQ-700)

**展示位置**: 统计看板顶部

**统计指标**:

| 指标名称 | 数据来源 | 说明 |
|---------|---------|------|
| 总软件包数 | software_packages 表 | 已发布状态的软件包总数 |
| 总订购数 | subscriptions 表 | 累计订购申请总数 |
| 活跃业务系统数 | business_systems 表 | 已启用的业务系统数量 |
| 本月新增订购 | subscriptions 表 | 当月新增的订购数 |

**交互**:
- 卡片展示环比变化（与上月对比）
- 点击可下钻到对应详情列表

### 3.2 订购量趋势图 (REQ-701)

**图表类型**: 折线图 + 柱状图（双轴）

**数据维度**:
- X轴: 时间（日/周/月）
- Y轴左: 订购数量
- Y轴右: 审批通过数量

**时间范围选项**:
- 近7天（默认）
- 近30天
- 近90天
- 近1年
- 自定义范围

**数据聚合规则**:
- 7天内：按日聚合
- 30天内：按日聚合
- 90天内：按周聚合
- 1年内：按月聚合

### 3.3 业务系统软件分布 (REQ-702)

**图表类型**: 饼图/环形图 + 明细表格

**展示维度**:
- 各业务系统的软件包使用数量分布
- TOP 10 业务系统（数量多时）

**数据计算**:
```sql
-- 按业务系统统计订购的软件包数量
SELECT business_system_id, COUNT(DISTINCT software_package_id) as pkg_count
FROM subscriptions
WHERE status = 'APPROVED'
GROUP BY business_system_id
```

**交互**:
- 点击饼图扇区下钻到该业务系统的软件列表
- 表格展示业务系统名称、系统编码、软件数量、占比

### 3.4 软件包热度排行 (REQ-703)

**图表类型**: 横向柱状图（TOP 10/20/50）

**排序维度**:
- 按订购次数（默认）
- 按订购的业务系统数（去重）
- 按浏览次数（如已记录）

**展示字段**:
- 排名
- 软件包名称
- 软件类型（图标区分）
- 订购次数 / 业务系统数
- 趋势箭头（与上期对比）

**交互**:
- 支持切换排序维度
- 点击软件包名跳转详情

### 3.5 软件类型分布 (REQ-704)

**图表类型**: 饼图 + 计数卡片

**展示内容**:
- 各软件类型的数量分布（Docker/Maven/NPM/PyPI/Generic）
- 各类型的订购热度对比

### 3.6 版本使用情况 (REQ-705)

**图表类型**: 堆叠柱状图

**展示内容**:
- 各软件包的版本分布
- 各版本的使用业务系统数
- 识别多版本并行使用的情况

## 4. 接口需求

### 4.1 API 列表

| 接口 | 方法 | 路径 | 说明 |
|------|------|------|------|
| 获取统计概览 | GET | `/api/v1/statistics/overview` | 返回卡片统计数据 |
| 获取趋势数据 | GET | `/api/v1/statistics/trend` | 返回趋势图数据 |
| 获取业务系统分布 | GET | `/api/v1/statistics/business-distribution` | 返回饼图数据 |
| 获取软件热度排行 | GET | `/api/v1/statistics/popularity` | 返回排行数据 |
| 获取类型分布 | GET | `/api/v1/statistics/type-distribution` | 返回类型分布 |
| 导出报表 | GET | `/api/v1/statistics/export` | 导出 Excel/PDF（管理员） |

### 4.2 接口详细定义

#### GET /api/v1/statistics/overview

**请求参数**: 无

**响应示例**:
```json
{
  "code": 200,
  "data": {
    "totalPackages": 45,
    "totalSubscriptions": 128,
    "activeBusinessSystems": 12,
    "newSubscriptionsThisMonth": 15,
    "trends": {
      "totalPackagesChange": 5.2,
      "totalSubscriptionsChange": 12.8,
      "activeBusinessSystemsChange": 0,
      "newSubscriptionsThisMonthChange": -3.2
    }
  }
}
```

#### GET /api/v1/statistics/trend

**请求参数**:
| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| days | Integer | 否 | 时间范围天数，默认7，可选7/30/90/365 |
| startDate | String | 否 | 自定义开始日期（yyyy-MM-dd） |
| endDate | String | 否 | 自定义结束日期（yyyy-MM-dd） |

**响应示例**:
```json
{
  "code": 200,
  "data": {
    "period": "daily",
    "startDate": "2026-03-18",
    "endDate": "2026-03-25",
    "data": [
      {
        "date": "2026-03-18",
        "subscriptionCount": 5,
        "approvedCount": 3,
        "rejectedCount": 1
      },
      ...
    ]
  }
}
```

#### GET /api/v1/statistics/popularity

**请求参数**:
| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| limit | Integer | 否 | 返回数量，默认10，最大100 |
| sortBy | String | 否 | 排序维度：subscription_count（默认）/business_system_count |

**响应示例**:
```json
{
  "code": 200,
  "data": [
    {
      "rank": 1,
      "packageId": 1,
      "packageName": "MySQL",
      "packageKey": "mysql",
      "softwareType": "DOCKER_IMAGE",
      "subscriptionCount": 25,
      "businessSystemCount": 8,
      "trend": "up"
    }
  ]
}
```

## 6. 数据存储需求

### 6.1 现有数据表复用

统计主要基于以下现有表进行聚合查询：

| 表名 | 用途 |
|------|------|
| software_packages | 软件包基础信息、状态 |
| subscriptions | 订购数据、审批状态 |
| business_systems | 业务系统信息 |
| users | 用户信息 |

### 6.2 统计快照表（可选优化）

对于大数据量场景，可考虑创建统计快照表：

```sql
-- 每日统计快照
CREATE TABLE statistics_daily (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    stat_date DATE NOT NULL,
    new_packages INT DEFAULT 0,
    new_subscriptions INT DEFAULT 0,
    approved_subscriptions INT DEFAULT 0,
    active_business_systems INT DEFAULT 0,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    UNIQUE KEY uk_date (stat_date)
);
```

## 7. 性能需求

### 7.1 响应时间

- 概览数据: < 500ms
- 趋势数据: < 1s
- 排行数据: < 500ms
- 图表数据: < 1s

### 7.2 优化策略

1. **索引优化**: 确保 subscriptions 表有 (created_at, status) 联合索引
2. **缓存策略**: 概览数据缓存5分钟，趋势数据缓存1小时
3. **分页聚合**: 大数据量时先聚合再分页
4. **异步加载**: 各图表独立加载，避免阻塞
