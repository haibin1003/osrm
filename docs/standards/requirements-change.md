# 需求变更管理规范

本规范定义 OSRM 项目的需求变更管理流程，确保变更的可追溯性，以及需求、设计、代码、测试之间的一致性。

---

## 文档信息

- **作者**: OSRM 开发团队
- **创建时间**: 2026-03-17
- **最后更新**: 2026-03-17
- **维护责任人**: 产品经理、技术负责人

### 字段说明

| 字段 | 说明 |
|------|------|
| **时间** | 变更提出的日期 |
| **变更内容** | 简要描述变更的内容 |
| **变更原因** | 为什么需要变更 |
| **影响范围** | 受影响的模块或功能 |
| **处理人** | 负责处理变更的人员 |
| **状态** | 待处理/进行中/已完成/已拒绝 |

## 变更流程

### 流程图

```
┌─────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│ 提出变更 │ → │ 影响评估  │ → │ 变更评审  │ → │ 实施变更  │ → │ 验证变更  │
│         │    │          │    │          │    │          │    │          │
└─────────┘    └──────────┘    └──────────┘    └──────────┘    └──────────┘
                    │               │
                    ↓               ↓
              ┌──────────┐    ┌──────────┐
              │ 拒绝变更  │    │ 退回修改  │
              └──────────┘    └──────────┘
```

### 详细步骤

#### Step 1: 提出变更

**发起人**：产品经理、开发人员、测试人员或业务方

**动作**：
1. 在需求文档「变更记录」章节添加变更记录，状态为「待处理」
2. 创建变更任务单（如使用 Jira、禅道等工具）
3. 通知相关干系人

#### Step 2: 影响评估

**负责人**：技术负责人

**评估内容**：
- 对现有功能的影响
- 对设计文档的影响
- 对代码的影响范围
- 对测试的影响
- 工作量评估
- 风险识别

**输出**：变更影响评估报告

#### Step 3: 变更评审

**参与人**：产品经理、技术负责人、开发代表、测试代表

**评审内容**：
- 变更的必要性
- 技术可行性
- 工作量合理性
- 风险可控性

**结果**：
- 批准 → 进入实施阶段
- 拒绝 → 记录拒绝原因
- 退回修改 → 补充信息或调整方案

#### Step 4: 实施变更

**负责人**：开发团队

**实施步骤**：
1. 更新需求文档
2. 更新设计文档（如需要）
3. 修改代码
4. 更新/补充测试
5. 更新功能总结文档

**注意**：变更实施必须遵循开发流程规范的五阶段流程。

#### Step 5: 验证变更

**负责人**：测试人员

**验证内容**：
- 变更是否正确实现
- 是否引入新的缺陷
- 相关功能是否正常

**输出**：测试报告

## 一致性检查机制

### 追溯关系图

```
需求文档 (REQ-XXX)
    ↓ 引用
设计文档 (引用 REQ-XXX)
    ↓ 引用
测试代码 (// 关联需求: REQ-XXX)
    ↓ 引用
实现代码 (// 关联需求: REQ-XXX)
    ↓ 引用
功能总结 (引用 REQ-XXX)
```

### 不一致场景处理

| 不一致场景 | 发现方式 | 处理流程 |
|-----------|----------|----------|
| **需求变更导致代码不一致** | 代码审查、测试失败 | 1. 创建变更任务<br>2. 按变更流程更新代码和测试<br>3. 更新设计文档 |
| **编码实现与需求不符** | 功能测试、代码审查 | 1. 对比需求和实现<br>2. 若实现有误：回退修改代码<br>3. 若需求需调整：发起需求变更评审 |
| **测试覆盖与需求不符** | 覆盖率报告、代码审查 | 1. 分析未覆盖的需求场景<br>2. 补充测试用例<br>3. 确保覆盖所有验收标准 |
| **设计文档与实现不符** | 代码审查、架构评审 | 1. 评估以哪个为准<br>2. 若设计过时：更新设计文档<br>3. 若实现偏离：重构代码 |

### 一致性检查工具

#### 1. 需求编号检查脚本

```bash
# 检查代码中是否引用了不存在的需求编号
# 检查需求文档中是否有过期的编号
```

#### 2. 代码审查检查项

在代码审查时必须检查：
- [ ] 代码注释中引用的需求编号是否存在
- [ ] 测试用例是否覆盖了需求的所有验收标准
- [ ] 实现逻辑是否与需求描述一致

#### 3. 自动化检查（CI）

在 CI 流程中添加：
- 需求编号有效性检查
- 代码与需求关联性检查

## 变更分类

### 按紧急程度

| 类型 | 定义 | 处理时限 |
|------|------|----------|
| **紧急变更** | 影响系统可用性或安全性的变更 | 24小时内处理 |
| **常规变更** | 正常的功能调整或优化 | 按迭代计划处理 |
| **低优先级变更** | 不影响当前迭代的优化建议 | 纳入后续迭代规划 |

### 按影响范围

| 类型 | 定义 | 审批层级 |
|------|------|----------|
| **小规模变更** | 影响单个模块，工作量 < 1 天 | 技术负责人审批 |
| **中规模变更** | 影响多个模块，工作量 1-3 天 | 技术负责人 + 产品经理审批 |
| **大规模变更** | 影响系统架构，工作量 > 3 天 | 技术评审委员会审批 |
