本规范定义 OSRM 项目的代码审查流程和标准，确保代码质量和一致性。

## 审查流程

### 流程图

```
┌─────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│ 提交代码 │ → │ 自动审查  │ → │ 人工审查  │ → │ 合并代码  │
│ (PR)   │    │ (code-reviewer)│    │ (可选)   │    │          │
└─────────┘    └──────────┘    └──────────┘    └──────────┘
                      │
                      ↓ 未通过
                ┌──────────┐
                │ 修复问题  │
                └──────────┘
```

### 详细步骤

#### Step 1: 提交代码（PR）

**要求**：
- PR 标题遵循规范：`[模块] 功能描述`
- PR 描述包含：
  - 关联的需求编号
  - 变更内容说明
  - 测试情况
  - 截图（UI 变更）

#### Step 2: 自动审查（code-reviewer 代理）

**触发方式**：
```
使用 code-reviewer 代理进行自动审查
```

**审查内容**：
- 代码风格和规范
- 潜在的安全问题
- 性能问题
- 可维护性问题

#### Step 3: 人工审查（可选）

**以下情况需要人工审查**：
- 架构性变更
- 核心业务逻辑修改
- 安全相关代码
- 复杂算法实现

**审查响应时间要求**：
| 优先级 | 响应时间 |
|--------|----------|
| 高 | 4 小时内 |
| 中 | 24 小时内 |
| 低 | 48 小时内 |

#### Step 4: 合并代码

**合并条件**：
- [ ] 自动审查通过
- [ ] 人工审查通过（如需要）
- [ ] 所有 CI 检查通过
- [ ] 代码覆盖率 ≥80%

## 审查清单

### 功能性检查

- [ ] **需求一致性**：功能实现是否符合需求文档
- [ ] **边界条件**：边界情况是否处理
- [ ] **错误处理**：错误场景是否有适当处理
- [ ] **业务逻辑**：业务逻辑是否正确

### 测试检查

- [ ] **测试覆盖**：测试覆盖率是否 ≥80%
- [ ] **测试质量**：测试用例是否有意义
- [ ] **TDD 遵循**：是否遵循 RED-GREEN-REFACTOR 流程
- [ ] **需求关联**：测试和代码是否标注需求编号

### 代码风格

- [ ] **命名规范**：类、方法、变量命名是否规范
- [ ] **代码格式**：代码格式是否符合规范
- [ ] **注释**：关键逻辑是否有适当注释
- [ ] **文档**：公共 API 是否有文档

### 性能

- [ ] **算法复杂度**：算法复杂度是否合理
- [ ] **数据库查询**：SQL 查询是否优化
- [ ] **资源使用**：资源使用是否合理
- [ ] **缓存**：是否适当使用缓存

### 安全性

- [ ] **输入验证**：用户输入是否经过验证
- [ ] **SQL 注入**：是否使用参数化查询
- [ ] **XSS**：输出是否经过编码
- [ ] **敏感数据**：敏感数据是否加密存储
- [ ] **权限控制**：权限检查是否完整

### 可维护性

- [ ] **代码复杂度**：圈复杂度是否合理
- [ ] **重复代码**：是否存在重复代码
- [ ] **单一职责**：类/方法是否单一职责
- [ ] **可测试性**：代码是否易于测试

## 审查响应标准

### 评论分级

| 级别 | 说明 | 处理方式 |
|------|------|----------|
| **BLOCKER** | 严重问题，必须修复 | 修复后才能合并 |
| **CRITICAL** | 重要问题，强烈建议修复 | 修复后才能合并 |
| **MAJOR** | 主要问题，建议修复 | 建议修复，可协商 |
| **MINOR** | 次要问题，可选修复 | 可选修复 |
| **INFO** | 信息性建议 | 仅供参考 |

### 评论模板

```
[级别] 问题描述

详细说明：
{详细说明问题}

建议：
{改进建议}

参考：
{相关链接或规范}
```

**示例**：
```
[CRITICAL] 缺少输入验证

详细说明：
该方法直接使用了用户输入的参数，没有进行任何验证，可能导致安全问题。

建议：
添加参数校验，使用 @Valid 注解或手动验证参数。

参考：
- 规范文档：docs/standards/api-design.md
- 示例代码：src/main/java/.../UserService.java:45
```

## 常见问题及处理

### 1. 代码风格不一致

**处理**：
- 优先修复影响可读性的问题
- 使用 IDE 自动格式化
- 配置 EditorConfig 统一风格

### 2. 缺少测试

**处理**：
- 要求补充单元测试
- 关键路径必须覆盖
- 覆盖率不达标禁止合并

### 3. 需求理解偏差

**处理**：
- 对照需求文档确认
- 如有歧义，产品经理介入
- 必要时发起需求变更

### 4. 性能问题

**处理**：
- 提供性能测试数据
- 评估影响范围
- 复杂优化可创建后续任务
