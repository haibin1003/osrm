# 文档编写规范

本规范定义 OSRM 项目的文档格式、模板和版本控制要求，确保文档的一致性和可维护性。

---

## 文档信息

- **作者**: OSRM 开发团队
- **创建时间**: 2026-03-17
- **最后更新**: 2026-03-17
- **维护责任人**: 技术负责人

---

## 文档格式

### 统一使用 Markdown

所有文档必须使用 Markdown 格式编写，文件扩展名为 `.md`。

### 文档头部信息

每个文档必须在开头包含以下元信息：

```markdown
# 文档标题

文档简要描述。

---

## 文档信息

- **作者**: 作者姓名
- **创建时间**: YYYY-MM-DD
- **最后更新**: YYYY-MM-DD
- **维护责任人**: 责任人姓名

---
```

### 正文格式规范

| 元素 | 规范 |
|------|------|
| **标题** | 使用 `#` 层级，一级标题只有一个（文档标题） |
| **列表** | 无序列表使用 `-`，有序列表使用 `1.` |
| **代码块** | 使用 ``` 包裹，标注语言类型 |
| **表格** | 用于结构化数据，包含表头 |
| **链接** | 使用相对路径链接到仓库内文档 |
| **图片** | 统一存放在 `docs/assets/` 目录，使用相对路径引用 |

---

## 版本控制

### Git 管理

- 所有文档必须纳入 Git 版本管理
- 关键文档（需求、设计）变更需通过 PR 流程
- 文档提交遵循相同的提交信息规范

### 文档提交规范

```
docs: 更新需求文档，增加审批流程说明

docs: 修正 API 设计文档中的错误参数

docs: 添加用户认证功能总结
```

## 文档存储规范

### 目录结构

```
/docs/
├── README.md                      # 文档目录导航
├── requirements/                  # 需求文档
│   ├── osrm-requirements.md       # 完整产品需求
│   └── features/                  # 按功能拆分的需求文档
│       ├── user-auth.md
│       └── software-package.md
├── design/                        # 设计文档
│   ├── architecture/              # 架构设计
│   │   └── user-auth-architecture.md
│   ├── database/                  # 数据库设计
│   │   └── user-auth-db.md
│   └── api/                       # API 设计
│       └── user-auth-api.md
├── standards/                     # 开发规范
│   ├── development-process.md
│   └── testing.md
└── summaries/                     # 功能总结文档
    └── user-auth-summary.md
```

### 文件命名规范

| 文档类型 | 命名规则 | 示例 |
|----------|----------|------|
| 需求文档 | `{feature-id}-{feature-name}.md` | `user-auth.md` |

| 数据库设计 | `{feature-id}-db.md` | `user-auth-db.md` |
| API 设计 | `{feature-id}-api.md` | `user-auth-api.md` |
